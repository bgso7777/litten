### 1. 영상 상세 팝업 본문 무한 로딩 버그 수정
- **증상**: 노트 > 영상구독 > 채널 펼침 > 영상 제목 탭 → 팝업의 헤더(채널명/제목/일시)는 정상이지만 본문이 영원히 로딩 스피너만 표시. 서버 로그도 안 찍힘.
- **Root cause**: `api_service.dart:976`의 `getYoutubeVideoDetail` URL이 `'$baseUrl/note/v1/youtube/videos/$videoId'` 로, **`/litten` context-path 누락** (다른 모든 엔드포인트는 `/litten/note/v1/...`). 404 → 서버 컨트롤러 미도달.
- **2차 버그**: `youtube_video_detail_dialog.dart`의 `_pollTimer`가 `detailCache.containsKey(videoId)`만 확인 → API 실패 시 cache에 안 들어가서 폴링이 영원히 지속됨.
- **수정 방안**:
  1. `api_service.dart:976` URL에 `/litten` prefix 추가
  2. `youtube_video_detail_dialog.dart` 폴링 종료 조건에 `!loadingSet.contains(videoId)` 추가 (실패 안전망)

### 4. 디바이스 간 리튼 메타데이터 동기화 (Phase 1) — **여기서 중단됨**
사용자가 짚은 문제: `Litten.id`는 클라이언트에서 `Uuid().v4()`로 생성되므로 디바이스 A/B가 같은 계정이어도 다른 littenId를 만든다 → `cloud_file` 경로 `{memberId}/{littenId}/{fileType}/{fileName}`이 디바이스마다 달라져 동기화 불가.

설계 합의:
- **방향 B 채택**: 파일 경로는 그대로 두고 **`cloud_litten` 테이블 추가하여 리튼 메타데이터 자체를 sync**. 디바이스 A의 littenId가 그대로 B로 전파됨.
- 충돌 정책: `updated_at_local` 기준 **Last-Write-Wins**.
- 기존 회원의 로컬 리튼은 첫 sync 시 **자동으로 서버 업로드** (다른 디바이스에서 바로 보이게).

Phase 1 작업 plan (다음 세션에서 진행):

- `cloud_litten` 테이블: `id PK / member_id / litten_id / title / description / created_at_local / updated_at_local / is_deleted / deleted_at` + BaseEntity 컬럼. `UNIQUE(member_id, litten_id)`, `INDEX(member_id, is_deleted)`, `INDEX(member_id, update_date_time)`

백엔드 (CloudFile 패턴 그대로):

- `GET    /note/v1/littens?since=<iso>` — 변경분 또는 전체
- `POST   /note/v1/littens` — upsert (LWW: 들어온 `updatedAtLocal`이 DB 것보다 최신이면 갱신)
- `DELETE /note/v1/littens/{littenId}` — soft delete + `deleted_at` 기록

프론트엔드:

- `services/litten_sync_service.dart` (신규) — 로그인 직후 풀 sync 1회 + 리튼 변경 즉시 push, 다운로드 시 LWW 머지(로컬에 없는 littenId는 새로 만들고, 있는데 updatedAt이 더 최신이면 덮어쓰기)

1. **무한 로딩 — 서버 로그도 안 찍힘**
   - 진단: `/litten` prefix 누락으로 404. 다이얼로그 폴링이 cache만 확인해 영원히 도는 2차 버그도 동반.

2. **iOS release 빌드 명령에서 cwd 누락**
   - 증상: 두 번째 빌드 명령이 `No pubspec.yaml found`로 실패했지만 `install`은 캐시된 직전 산출물로 성공해 사용자가 의도한 산출물이 아니었음.

3. **Maven 컴파일 검증 실패**
   - 증상: 로컬 JDK 환경에서 Lombok 호환성 문제로 `mvn compile` 실패.

### 얻은 교훈
- "git status clean"이 **곧 변경 없음을 의미하지 않음** — 새 커밋으로 정리되면 워킹트리는 깨끗해진다. 세션 중 사용자가 직접 커밋했을 가능성을 고려해 `git log --oneline -N` 을 함께 확인하는 습관 필요.
- 백그라운드 빌드 명령의 cwd는 보장되지 않는다. 첫 빌드는 우연히 동작했지만 두 번째는 실패. **장시간 명령은 항상 명시적 cd**.
- 사용자가 "임시로 만들어 주고"라고 해도 데이터 모델 결정(단일 vs 다중 선택, type 이름)은 분명히 정해야 변경이 커지지 않는다. 진행 전 짧은 AskUserQuestion 1~2개가 효율적.

### 미래 개발자를 위한 팁
- **백엔드 새 엔드포인트 작성 시**: URL은 `/note/v1/...` 로 시작하지만 클라이언트 호출은 `$baseUrl/litten/note/v1/...` 임을 잊지 말 것. `ApiService`에 새 엔드포인트 상수를 추가해 일관성 유지 권장.
- **다이얼로그/시트가 부모 setState로 리빌드되지 않는 케이스**: 폴링 종료 조건은 *성공 캐시 채워짐* 외에 *로딩 끝났음(실패 포함)* 도 함께 보도록 작성하면 무한 로딩을 막을 수 있음.
- **JPA save와 dirty checking**: `noteMemberRepository.save(noteMember)` 가 모든 컬럼을 다시 쓰는 것이 아니라 변경된 필드만 UPDATE. 패스워드 같은 민감 컬럼은 의도치 않게 덮어쓸 걱정 없음.
- **로컬에서 백엔드 컴파일 검증이 어렵다면** 시그니처/필드 추가 같은 단순 변경만 한정해 사용자 배포 환경에서 검증하도록 위임. 그러나 복잡한 로직 변경은 사용자 빌드 결과를 받아 확인 필요.
- **다음 세션 시작 시**: 이 문서의 "4. 디바이스 간 리튼 메타데이터 동기화 (Phase 1)" 섹션을 그대로 작업 plan으로 사용. `CloudFile*` 클래스들을 참고 패턴으로 미러링하면 빠르게 작성 가능.
