iOS는 `build/ios`에 남은 1.8.0 산출물(`NaverThirdPartyLogin.h` 참조)이 걸려 해당 디렉토리 삭제 후 재빌드로 해결. (2.1.1의 iOS는 순수 Swift라 그 헤더가 없음)

**2. "지운 룸이 그룹 룸 목록에 계속 보임" → 삭제된 적이 없었음**
룸 목록의 버튼은 `방 나가기`(`hideConversation`, 로컬 숨김)이고 `deleteShareGroup`(서버 soft delete)은 그룹 관리 다이얼로그의 휴지통에서만 호출됐다. 백엔드는 `isDeletedFalse`로 정상 필터링 중 → **서버 기준으로는 정상 동작**이었고 UX 문제였다.
해결: 방장에게만 `그룹 삭제` 선택지 노출(모든 멤버에게서 삭제, 되돌릴 수 없음 명시).

**3. "자료 삭제 버튼이 안 보임" → 방 종류를 오판했음**
해당 방(`리튼 개선`)이 그룹룸이 아니라 **셀프룸**(`note_self_study_room` id=4)이었다. 셀프룸 파일은 `shareId` 없이 `__selfFile:true`라 공유 자료 전용 조건에서 제외되어 버튼이 렌더링되지 않았다. 목록엔 똑같이 "보냄"으로 보여 구분이 안 됨.
셀프룸은 **항목 단위 삭제가 프론트·백엔드 모두 없었고**(방 전체 삭제만 존재) 신규 구현으로 해결.
※ 이 건은 확인 없이 "이미 구현되어 있다"고 답했다가 사용자 스크린샷으로 반박된 케이스.

### 구현 세부

**그룹룸 멤버 권한**
- `note_study_room` + `allow_member_chat`(기본 1) / `allow_member_file`(기본 0)
- 강제 지점: `RoomMessageService.send`, `RoomShareService.createShare` — 방장은 항상 통과, 멤버만 검사
- 생성 시 체크박스 2개 + 방장이 나중에 변경 가능한 `PATCH /study-rooms/{id}/options` 신설(낙관적 반영 후 실패 시 롤백)
- 구 데이터 NULL 방어: `!Boolean.FALSE.equals(chat)` / `Boolean.TRUE.equals(file)`
- 차단 안내는 **기존 스낵바 경로가 서버 메시지를 그대로 노출**해 프론트 추가 작업 불필요했음

**멤버 조회 검증**
- 기존 `GET /members/search`(비인증 가능, `{found,id,name}`)를 멤버 추가 경로에 연결
- 실서버 검증: 닉네임 `tom` → `bgso777@naver.com`으로 해석 확인 → **입력값이 아니라 조회 결과 `id`(이메일)를 키로 전달**하도록 구현(오타·표기 차이 흡수)
- 새 그룹 다이얼로그: 검증 통과분만 칩 생성(초록 체크 + `닉네임 (이메일)`), 입력창 잔여값도 생성 직전 검증 후 실패 시 생성 중단(기존엔 그대로 넘어가 서버에서 조용히 누락)

**자료 삭제**
- 공유 자료: 기존 백엔드 `cancel()` 활용 — 발신자 확인 → 공유 + 전 수신자 delivery soft delete + 실파일 삭제. `received()`가 삭제 delivery와 삭제 share를 **이중 필터링**해 타 멤버 목록에서도 사라짐을 코드로 확인
- UI: [자료] 탭에 삭제 버튼 신규(원래 삭제 동작 자체가 없었음) + 말풍선 `공유 취소`(길게 눌러야 접근·확인 없이 즉시 실행)를 `삭제`로 변경하고 **확인 다이얼로그** 추가
- 셀프룸: `SelfStudyRoomService.deleteItem()` + `DELETE /my-study-rooms/items/{itemId}`, 프론트 `deleteSelfChatItem()`(로컬 목록·복사본 파일·서버 항목 정리)

### 얻은 교훈

- **우회 패치가 쌓이면 근본 원인을 다시 보라.** 한 패키지에 Gradle 패치 3개는 신호였고, 버전 업그레이드가 정답이었다.
- **"이미 구현됨"이라 답하기 전에 사용자가 보는 화면과 같은 조건인지 확인하라.** 그룹룸/셀프룸을 구분하지 않아 오답을 냈고, DB 조회 한 번으로 즉시 판명됐다.
- **이 백엔드는 인증 없는 요청에 경로 유무와 무관하게 빈 200을 반환한다.** curl로 엔드포인트 존재 확인이 불가능하니, 배포 검증은 **배포된 .class 파일에서 심볼을 grep**하는 방식이 확실하다.
- 사용자 체감("삭제했는데 보인다")과 서버 상태(정상)가 다를 수 있다. 먼저 어느 쪽 문제인지 가른 뒤 손댈 것.
- `pgrep -f catalina`는 **자기 자신의 명령 문자열을 잡는다.** 종료 확인 시 `org.apache.catalina.startup.Bootstrap`으로 좁힐 것.

### 미래 개발자를 위한 팁

- **커밋·AOS 에뮬 설치는 사용자가 수동 처리**한다(아이폰 실기기 설치는 예외로 진행). 이번 작업분 26개 파일이 미커밋 상태이니 다음 세션에서 먼저 확인할 것.
- 아이폰 설치는 반드시 `flutter build ios --release` + `devicectl device install`. `flutter install`/`run`은 앱을 지워 **데이터가 초기화된다.**
- 서버 배포는 tar 금지(macOS AppleDouble `._*`가 Spring 컴포넌트 스캔을 깨 과거 13분 다운 발생). 단일 `.class` scp + `COPYFILE_DISABLE=1`.
- 백엔드 빌드는 `backend/build.sh` 사용(전역 JDK 25 그대로 두고 이 빌드만 JDK 17 = 서버 타겟, bytecode major 61).
- 로컬 DB(`litten7`)엔 `note_member` 계열 3개 테이블뿐이다. 스터디룸 등 실데이터는 **원격 DB에만** 있으니 스키마 확인·마이그레이션은 원격으로.
- 룸 종류가 3가지(그룹룸 / 1:1 / 셀프룸)이고 데이터 소스가 각각 다르다(`note_study_room` / 공유 / `note_self_study_room`). 자료 관련 기능은 **세 경우를 모두 확인**할 것 — 이번 세션의 오판 원인.
