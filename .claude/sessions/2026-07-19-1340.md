### 얻은 교훈

1. **가동 중인 서버에 배포 파일을 올리지 말 것** — autoDeploy가 개입해 예측 못한 상태가 됨
2. **빌드 성공 문구를 그대로 믿지 말 것** — 실제 컴파일 여부를 로그로 확인
3. **기기 설치 전 연결 방식을 확인할 것** — 이름만 보고 단정하면 엉뚱한 기기에 설치됨
4. **UI를 복제하면 반드시 어긋난다** — 이번에 6회에 걸쳐 수동으로 맞췄고 앞으로도 양쪽을 함께 고쳐야 함
5. **DB 순서가 코드보다 먼저** — 컬럼 없이 새 코드를 올리면 API 전체가 깨짐

### 미래 개발자를 위한 팁

- 백엔드 컴파일: `export JAVA_HOME=/opt/homebrew/opt/openjdk@17/...` 또는 `./build.sh` 사용
- 안드로이드 패키지명 `com.litten.app`, iOS 번들 ID `com.sobyungkyu.litten` — 서로 다름
- 아이폰 설치 시 데이터 유지는 `xcrun devicectl device install app` 사용 (flutter install은 데이터 초기화)
- 새 테스트 기기는 `xcodebuild -allowProvisioningDeviceRegistration`으로 자동 등록 가능
- iOS 시뮬레이터는 CLI 터치 수단이 없어 UI 검증은 사용자 확인이 필요
- 디스크가 자주 100%에 도달함 — DerivedData와 frontend/build는 언제든 지워도 무방
- 셀 일정 관련 코드 위치: 백엔드 `studyroom/schedule/`, 프론트 `home_dashboard_screen.dart`(다이얼로그·카드), `schedule_utils.dart`(반복 계산), `litten_unified_list_view.dart`(목록 타일)
